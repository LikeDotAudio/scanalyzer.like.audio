Instant long-file previews: a binary peak map inside the .PEAK

Problem: long files are painful — the analyzer skips anything over 600 s outright, and the UI decodes the entire file before drawing a single waveform pixel. Design: an 8-bit min/max peak map, computed in one streaming pass, stored base64 inside the .PEAK, painted in the next frame. Companion prose: binary_peak_preview_audit.md.

The schema

"preview": {
  "preview_version": 1,
  "samples_per_bin": 16384,        // smallest power of two so bin_count ≤ 16384 (floor 256)
  "bin_count": 10547,
  "bits_per_value": 8,             // signed, interleaved min,max per bin
  "channel_mode": "mono_mixdown",
  "peak_data_base64": "…"          // ≤ ~43 KB worst case, any file length
}

Inside the .PEAK — not a second binary file (+1 open × 40 k files), not a chunk written into the user's audio. The manifest's projectSlim() never copies it, so startup stays slim; the Examiner already fetches the full record on select, so the preview arrives on a read the app already pays for.

Three paths

WRITE (new scans) decode pass ──▶ fold min/max per bin (O(n), noise vs the STFT) ──▶ preview into the record ≤ 600 s: full DSP as today · > 600 s: preview_only — metadata + preview + peak/RMS, streaming fold over symphonia packets, PCM never materialized — constant memory APPEND (backfill) walk sidecars ──▶ missing preview? ──▶ stream-decode audio (no DSP, IO-bound) ──▶ read .PEAK as serde_json::Value (typed structs DROP unknown fields — data loss) ──▶ merge preview ──▶ write .PEAK.tmp ──▶ atomic rename ──▶ rebuild manifest READ (select) record already fetched ──▶ atob → Int8Array (µs) ──▶ paint next frame audio element streams immediately ──▶ full decode demoted to background ──▶ swaps in for deep zoom / spectrogram / region detect when ready

Two principles carry the whole design: piggyback small data on reads you already pay for — the expensive unit is the file open, not the bytes — and store at the fidelity of the consumer: a ≤ 400 px canvas can show 8 bits; everything above that is decode time, not information.
